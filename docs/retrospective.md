## 2. 缺口与断层分析 (Gap Analysis)

### G01 — 缺少架构规划阶段

**缺失项**: 任何形式的架构设计文档、数据流图、模块边界定义。

**实际发生路径**:
```
用户需求 → 直接写代码 → 下一个需求 → 继续堆叠 → 模块间耦合累积 → 重构成本 > 重写成本
```

**推导依据**: Git 历史仅 2 个 commit，说明所有功能是一次性或短时间连续提交完成，没有中间评审或重构节点。

### G02 — 缺少变更影响评估

**缺失项**: 每个新功能引入前对其与现有模块交互方式的评估。

**案例 — LLM 集成（致命断层）**:

| 引入变更 | 影响模块 | 实际后果 |
|---------|---------|---------|
| `llm-service.js` | `server/index.js` | `broadcast()` 函数增加了异步 LLM 调用和 `await`，将非关键路径混入主推送链路 |
| `health:update` 消息 | `client/app.js` → `visualizer.js` | UI 需要处理额外的消息类型，增加了 switch 分支复杂度 |
| 50+ 工程规则 | `project-analyzer.js` | 规则扫描与树构建耦合，全量分析耗时从 <10ms 暴涨到 >200ms |
| 违规检测 | `visualizer.js` | 节点需要渲染额外的违规标记，更新逻辑扩散 |

**推导依据**: LLM 集成属于 `P2（扩展功能）`，但实际实现时与 `P0（核心链路）` 耦合，未做功能隔离。

### G03 — 过度设计 (Over-engineering)

**具体表现**:

| 过度设计 | 实际使用 | 维护成本 |
|---------|---------|---------|
| F13 三层描述系统（L1 硬编码 / L2 用户自定义 / L3 LLM 生成 / L4 社群共享） | 仅 L1 有实际数据，L2-L4 在 72h 内从未触发 | 查询优先级逻辑（L3→L1→L2）增加了不必要的复杂度 |
| 异步 LLM 批处理缓存系统（.llm-descriptions-cache.json） | 项目仅 7 个文件，硬编码描述完全覆盖 | 缓存文件管理 + 缓存失效逻辑 + 格式校验 |
| 健康评分（0-100 四色体系 + 增长动画） | 评分规则由开发者主观定义，无用户反馈校准 | 额外 API 端点 + WebSocket 消息 + UI 渲染 + 动画帧 |
| 违规检测（50+ 规则逐条扫描） | 规则的差异化权重和优先级从未使用 | 全量扫描 + 结果聚合 + 前端过滤 + 展示 |

**推导依据**: 用户的实际需求是"看清 AI Agent 在改什么"，但系统走向了"AI 帮你分析 AI 改得好不好"——在错误的抽象层级上解决问题。

### G04 — D3 数据绑定不一致（未修复 Bug）

**具体表现**: 边渲染断裂，部分节点间连线在 DOM 中呈现为 `M0,0L0,0`（即起点终点重合）。

**根本原因**（基于代码分析）:

```
visualizer.js 中 forceLink 的 source/target 引用问题:

1. 首次渲染: this.edges 中的 source/target 为字符串（文件路径）
2. D3 forceLink 初始化时会将这些字符串解析为 simulation.nodes() 中的对象引用
3. 增量更新 (incrementalUpdate) 时:
   - 新 edge 加入 → source/target 可能是对象引用或字符串，混合状态
   - 节点重绑定 → 部分 node 对象被替换 → edge 引用的对象指针失效
   - 结果: 边的 source/target 指向了不在 simulation.nodes() 中的对象 → D3 退化为 M0,0L0,0
```

**影响链**: 力导向图是核心 P0 功能 → 边渲染断裂直接破坏核心认知 → 用户失去对依赖关系的基本信任 → 工具不可用。

**为什么没有被修复**: 每次增量更新都在已有代码上修补 → 修补引入新问题 → 开发者陷入"修一个冒两个"的恶性循环。根因是：没有建立 D3 数据绑定的一致性契约。

### G05 — 冷启动策略缺失

**缺失项**: 新用户首次打开项目时缺乏逐步引导。

**当前行为**: VS Code 风格的"空状态 + 按钮"——适合开发者但不适合 PRD 中定义的"纯小白 Vibe Coder"。

**推导依据**: index.html 中的空状态引导仅有图标 + "Open a project to start monitoring"，没有示例项目、没有 onboarding 步骤、没有首次使用弹窗。PRD（第 7 章降级策略）提到了空状态引导，但实际实现仅覆盖了基础要求。

## 3. 根因诊断 (Root Cause Analysis)

### R01 — 没有模块边界，只有文件

**症状**: 函数和类可以访问任何模块的内部状态。

```
server/index.js 的 broadcast() 函数:
  → 发送 WebSocket 消息（自己的事 ✓）
  → 触发 LLM 异步分析（侵入 server/llm-service.js ✗）
  → 广播 health:update（侵入 server/llm-service.js ✗）
  → 调用 analyzer.analyze()（侵入 project-analyzer.js ✗）

client/app.js 的 _handleFileChanged():
  → 更新 graph（自己的事 ✓）
  → 显示分析中 banner（侵入 UI 渲染 ✗）
  → 触发 LLM 分析（侵入 server 职责 ✗）
```

**修正方案**: 单一定义每个模块的"允许做什么"和"禁止做什么"。

### R02 — 异步链路侵入同步渲染

**症状**: 核心渲染路径（WebSocket → D3 更新）与 LLM 异步路径共享同一通道。

```
WebSocket 消息到达
  → broadcast() 发送给所有客户端 ✓
  → broadcast() 内部触发 LLM 调用 ✗（LLM 调用是异步且可能失败的）
  → LLM 调用返回后通过 WebSocket 推送 health:update ✗（与文件事件竞争）
  → 客户端收到 health:update 触发重渲染 ✗（graph 可能在动画中）
```

**问题本质**: P0 链路和 P2 链路没有物理隔离。理想做法是 LLM 异步路径使用独立的消息通道或轮询机制。

### R03 — AI 功能设计过度

**问题链**:
```
"AI 能生成语义描述" 
  → 需要缓存系统管理 AI 输出
  → 缓存需要失效策略
  → 失效需要检测文件变更比例 > 10%
  → 检测触发全量分析
  → 全量分析消耗 200ms+
  
实际上 7 个文件的硬编码描述（F13）已经满足需求。
```

**代价**: 为"可能有用"的功能支付了"确定的复杂度"。

### R04 — 没有灰度发布或功能开关

**症状**: 所有功能默认启用且不可关闭。

- 健康评分功能从实现的第一秒就对所有用户可见
- LLM 分析在首次打开项目时自动触发（即使没有 API Key，会静默失败）
- 编辑动画对所有节点生效（无法关闭）

**修正**: 核心 P0 默认启用，P1/P2 功能按需激活或通过配置开关。

### R05 — 代码量增长失控

| 指标 | P0 阶段 | P3 峰值 | 清理后 |
|------|---------|---------|--------|
| 源文件数 | 7 | 8 | 7 |
| 总 LOC | ~1800 | ~4500 | ~3424 |
| CSS LOC | ~400 | ~1256 | ~995 |
| visualizer LOC | ~600 | ~1500 | ~1364 |
| API 依赖 | 0 | 1 (LLM) | 0 |

72 小时内代码量膨胀 2.5 倍。CSS 膨胀至 1256 行（其中 132 行是活动横幅，从未被用户提及需求）。

## 4. 具体失败案例 (Case Studies)

### Case 1: package.json 假编辑状态

**用户报告**: "没有 agent 在修改为什么一直显示在被编辑？"

**根因**: macOS fsevents 的 `raw` 事件会因 metadata 变更（如 Spotlight 索引、Finder touch）触发。编辑器保存时也会先写临时文件再 rename，chokidar 可能触发多次事件。

**修复**: 移除 `raw` 事件监听，只监听 `change` 事件（文件内容实际写入）。

**教训**: 文件系统事件的语义在不同 OS 上不同。`change` ≠ `modified metadata`。在选择事件源时需要理解 OS 层行为。

### Case 2: 边缘动画过度设计

**演变路径**:
```
v1: 编辑节点缩放 1.4 倍 + 0.4s cubic-bezier 过冲效果 → 用户"太不优雅了"
v2: 呼吸动画（2.2s ease-in-out，缩放 1.04）→ 仍过于明显
v3: 静态橙色填充 → 用户接受
```

**教训**: 编辑指示的核心需求是"告诉用户哪个文件正在被改"，不是"展示流畅的 CSS 动画"。最简单的方案（颜色变化）效果最好。每次动画迭代都增加了 CSS 和渲染复杂度，但用户只需要一个信号。

### Case 3: D3 边断裂未修复

**用户报告**: "很多线路都是断的，没有开头没有结尾，很多节点也没有连线，但是聚焦下又会显示和他有关联的节点"

**直接原因**: focus mode 使用 `this.edges`（数据层）查找邻居，而 SVG 渲染使用 D3 force 的 tick handler（渲染层）。当数据绑定中的 source/target 引用失效，渲染层无法绘制路径。

**未修复原因**: 
- 首次修复尝试改了错误位置（误判为 tick handler closure 问题）
- 二次跟踪发现数据层问题，但修复范围不明确
- 增量更新机制（app.js 的 500ms batching + visualizer 的 6 种事件类型）使得追踪数据流变得极其复杂

**教训**: 可视化工具的"数据-状态-渲染"三层的同步关系必须在架构层面定义，不能依赖隐式 D3 行为。

## 7. 纪律规则 (Engineering Discipline)

以下规则用于防止再次陷入"屎山"状态：

### D01 — 功能分层隔离

```
P0 (核心)  → server/ + client/ 基础模块
P1 (重要)  → 独立子目录或明确接口前缀
P2 (扩展)  → 功能开关保护，默认关闭
LLM 功能   → 独立进程/模块，不得侵入 broadcast() 等主链路

规则: P1/P2 功能必须通过明确的 import 或配置启用，不得默认激活。
```

### D02 — 代码量警戒线

| 指标 | 警戒线 | 超限措施 |
|------|-------|---------|
| visualizer.js | < 1500 LOC | 强制拆分渲染引擎 / 交互层 / 数据层 |
| app.js | < 600 LOC | 强制拆分消息处理 / 状态管理 / UI 控制 |
| styles.css | < 1200 LOC | 基建样式与功能样式分离文件 |
| 单个功能 PR | < 300 LOC diff | 需说明为何超过 |

### D03 — 新增功能检查清单

在向项目添加任何新功能前，必须自检：

- [ ] 这个功能解决的是用户报告的痛点（不是"可能有用"）？
- [ ] 它属于哪个优先级（P0/P1/P2）？
- [ ] 它需要多少新增代码？（预估 > 200 LOC 需要先做架构设计）
- [ ] 它是否依赖外部 API？（如果是，必须有降级方案）
- [ ] 它侵入哪些现有模块？（必须列出所有需要修改的文件）
- [ ] 如果没有这个功能，用户的核心体验是否受影响？
- [ ] 这个功能可以被用户关闭吗？

### D04 — 重构触发条件

以下任一条件满足时，必须先重构再添加新功能：

1. `visualizer.js` > 1500 LOC
2. `server/index.js` > 200 LOC
3. 任何文件存在超过 2 个未修复的开放 Bug
4. D3 边渲染有任何已知 BUG（当前条件满足！）
5. 测试项目无法一次性走通全链路

## 8. 结论

**一句话总结**: Vibe Guarding 在 72 小时内从"有用的监控工具"膨胀为"什么都想做但核心功能没做好的屎山"，根源不在于代码质量，而在于缺乏架构纪律和功能优先级管理。

**目前的清理状态**:
- 已移除所有 LLM 依赖（零配置启动）
- 已回滚到核心 P0 + 部分 P1 功能
- 已清理测试项目和冗余文档
- **D3 边渲染断裂尚未修复**——这是下一个开发 session 的第一个任务

**建议的下一个动作**: 不添加任何新功能，先修复 FIX-01 和 FIX-02，然后使用真实项目验证全链路稳定。在新功能通过 D03 检查清单前，不接受任何 P1/P2 功能的代码合并。
