# Vibe Guarding — PRD

## 1. 背景与问题定义 (Problem Statement)

### 替代的工作流

目前使用 AI Agent 进行 Vibe Coding 的开发者，依赖以下人工监督流程：
- 手动切换到文件总管查看新增了哪些文件
- 在 IDE 的 diff 视图中逐个文件审阅变更
- 凭记忆追踪项目结构的演变
- 手动将错误文件路径复制给 Agent

Vibe Guarding 将上述流程替代为：**一个即时更新的可视化界面，自动呈现项目结构的动态演变，并提供一键定位与反馈机制。**

### 核心痛点

| 痛点 | 现状描述 | 量化影响（估算） |
|------|---------|----------------|
| **结构迷失** | 开发者无法直观感知 Agent 正在操作哪个文件及其在整个架构中的位置 | 每次上下文切换损失 30-60 秒重新定位时间 |
| **屎山风险** | 无法即时监督 Agent 是否遵循工程化原则，容易产生耦合松散、缺乏分层的代码 | 后续重构成本增加 3-10 倍 |
| **调试成本高** | Bug 发生时需要手动搜索文件路径，再将错误上下文传递给 Agent | 每次调试额外消耗 200-500 Tokens 用于路径描述 |
| **PRD 偏离** | 开发过程中无法即时对比实际产出与 PRD 规划的差异 | 功能遗漏或范围蔓延，平均偏离 30-50% |
| **认知鸿沟** | 非技术用户无法理解“文件级”力导向图，缺乏业务语义翻译层 | 工具仅对开发者可用，小白用户流失率 > 60% |

## 2. 目标用户与使用场景

### 用户画像

| 角色 | 描述 | 核心需求 |
|------|------|---------|
| **Vibe Coder（主要）** | 使用 Cursor/Claude Code/Windsurf 等 AI 程序设计工具的开发者 | 即时掌握项目动态，确保 Agent 按规划产出 |
| **Tech Lead / Reviewer** | 需要审阅 Agent 产生代码的技术管理者 | 快速定位问题模块，降低审阅时间 |
| **纯小白 Vibe Coder** | 无程序设计基础，透过自然语言驱动 AI 编程的用户（如产品经理、设计师） | 用业务语言而非技术语言理解项目结构，追踪 AI 正在改哪里 |
| **AI Agent 自身** | 透过工具获取精确的文件路径与错误定位，减少 Token 浪费 | 获得精准的错误坐标而非模糊描述 |

### 触发场景

| 场景 | When | Where | Why |
|------|------|-------|-----|
| **Vibe Coding 中** | Agent 正在生成/修改代码时 | 开发者工作站的浏览器 | 即时监督 Agent 行为，避免偏离方向 |
| **Code Review 时** | Agent 完成一批变更后 | Review 前准备阶段 | 快速理解变更范围与影响模块 |
| **Debug 时** | 发现 Bug 需要定位 | 测试或验收阶段 | 一键复制错误路径给 Agent，精准调试 |
| **PRD 对齐检查** | 里程碑节点 | 项目管理回顾 | 确认实现进度与 PRD 规划的 gap |

## 4. AI 介入策略 (AI Integration Strategy)

### 模型调用位置与目的

Vibe Guarding 本身不直接调用 LLM，而是作为 **AI Agent 与开发者之间的可视化桥梁**。AI 介入体现在以下两个环节：

| 环节 | 目的 | 方式 |
|------|------|------|
| **PRD 解析** | 将自然语言 PRD 转换为结构化的 feature/module 清单 | 基于规则的解析器（无 LLM 调用），确保可预测性与零成本 |
| **Agent 提示生成** | 根据选取节点的文件路径与角色，产生精确的提问提示词 | 模板化拼接（无 LLM 调用），确保格式统一 |

### Prompt 策略概述

核心原则：**即时链路零 LLM，异步环节可引入 LLM。** 即时图形渲染与文件事件处理必须 100% 确定性，但非阻塞的后台任务可安全引入 LLM。

**即时链路（零 LLM）：**
1. 文件监控 → 图形更新：完全基于事件驱动，无 LLM 介入
2. PRD 解析：基于规则解析器，确保可预测性与零成本
3. Agent 提示生成：模板化拼接，确保格式统一

**异步环节（可引入 LLM）：**
1. 语义映射生成（F16）：项目首次载入时，后台触发批量 LLM 请求，为每个文件生成大白话描述
2. 此环节不阻塞 UI 渲染，允许 5-15 秒延迟，完成后更新 tooltip

### 异步 LLM 摘要策略（F16）

#### 架构

```
项目载入完成
  ↓ [触发条件：首次载入 / 文件结构变更 > 10%]
Server 收集文件树 + role 资讯
  ↓ [POST 到 LLM API]
LLM 返回 JSON { path → human_name, metaphor_desc }
  ↓ [写入本地缓存]
Client GET /api/project/human-descriptions
  ↓ [visualizer 在 tooltip 中渲染]
hover → 显示“👁️ 监控保安：盯着文件变化的程序”
```

#### Prompt 模板

```
# Role
你是一个极具耐心的编程导师，擅长用最通俗易懂的隐喻向完全没有编程基础的文科生解释软件工程。

# Task
我将提供一个软件项目的目录树结构以及部分核心文件的名称。
请你为每个关键文件/文件夹生成一段“大白话”描述，帮助纯小白理解它们的作用。

# Rules
1. 禁用一切技术黑话（如：路由、DOM、依赖注入、异步请求等）。
2. 使用生活化的隐喻（如：大脑、图纸、仓库、快递员、皮肤）。
3. 描述必须极简，每条不超过 20 个字。

# Output Format (Strict JSON)
{
  "mappings": [
    {"path": "client/index.html", "human_name": "网页骨架", "metaphor_desc": "用户看到的页面基础框架"},
    {"path": "server/file-watcher.js", "human_name": "监控保安", "metaphor_desc": "在后台死死盯着文件变化的程序"}
  ]
}
```

#### 降级策略

| 风险 | 降级方案 |
|------|---------|
| LLM API 不可用（无 Key/网络中断） | 静默降级到 F13 硬编码映射规则 |
| LLM 返回格式错误 | try/catch + 正则后备提取，失败则跳过该文件 |
| 大项目 Token 超限 | 只提交核心文件，过滤 node_modules/build，每批上限 100 个文件 |
| 延迟过高 | 异步进行，不阻塞 UI；若 30 秒未完成则超时放弃 |

## 5. 评测指标与验收标准 (Evals & Acceptance Criteria)

### 定量指标

| 指标 | 目标值 | 测量方式 |
|------|-------|---------|
| 文件事件延迟（P95） | < 500ms（从文件变更到 UI 更新） | 浏览器 Performance API 计时 |
| 图形渲染帧率 | > 30 FPS（1000 节点以内） | Chrome DevTools FPS meter |
| WebSocket 连接稳定性 | 断线后 3 秒内自动重连 | 手动断网测试 |
| 依赖解析准确率 | > 85%（JS/TS 项目） | 与 `madge` 工具输出比对 |
| 节点角色辨识准确率 | > 80% | 人工抽样 50 个文件验证 |
| PRD 解析召回率 | > 90%（正确提取 features） | 与人工标注比对 |

### 定性验收

| 场景 | 验收标准 |
|------|---------|
| 首次开启项目 | 5 秒内呈现完整图形，节点布局稳定 |
| 文件新增 | 新节点在 1 秒内出现在图形中，带入场动画 |
| 文件删除 | 对应节点在 1 秒内移除，相关边同步消失 |
| 节点点选 | 右侧面板在 100ms 内展示完整资讯 |
| Bug 标记 | 点选后节点立即变红，再次点选恢复 |
| PRD 载入 | ghost node 在 1 秒内出现在图形中 |
| 大量文件变更（批量新增 10+ 文件） | 图形在 3 秒内完成更新，不出现卡顿 |

## 6. 冷启动资料策略 (Cold Start Data Strategy)

### 初期资料来源

| 资料类型 | 来源 | 取得方式 |
|---------|------|---------|
| 项目文件结构 | 用户指定目录 | 启动时递归扫描文件系统 |
| 依赖关系 | 项目源代码 | 启动时全量扫描 import/require |
| 文件角色 | 项目源代码 | 基于扩展名与内容规则推断 |
| 修改时间 | 文件系统 metadata | `fs.statSync` 取得 mtime |

### 人话描述缓存策略（F13 / F16）

| 层级 | 来源 | 更新时机 |
|------|------|---------|
| L1 硬编码 | `project-analyzer.js` 中的规则映射表 | 随代码版本更新 |
| L2 用户自定义 | `~/.vibe-guarding/custom-descriptions.json` | 用户透过 UI 编辑后写入 |
| L3 LLM 生成（F16） | 异步请求 LLM API | 首次载入 / 文件结构变更 > 10% |
| L4 社群共享 | 可导出的 JSON 规则集 | 用户手动导入 |

查询优先级：L3 → L1 → L2（L3 为动态生成，每次载入可能不同；L1 为静态备份）

### 资料飞轮构建路径

```
Phase 1（目前）
  规则引擎 → 精确但有限 coverage

Phase 2（用户反馈校正）
  用户修正角色标签 → 存入本地规则缓存
  → 下次扫描自动套用已确认规则

Phase 3（社群规则贡献）
  共享角色辨识规则集（JSON 格式）
  → 根据项目类型（React / Vue / Django）载入对应规则
```

## 7. 降级与兜底策略 (Fallback Strategy)

| 风险项 | 描述 | 降级方案 |
|--------|------|---------|
| **WebSocket 断线** | 浏览器与后端 Server 的长连接中断 | 自动重连机制（指数退避 1s/2s/4s，max 30s），断线期间缓存事件，重连后批量发送 |
| **文件监控失败** | chokidar 因权限或 inotify 限制无法监听 | 降级为轮询模式（每 2 秒扫描一次），提示用户调整目录权限 |
| **D3.js 效能瓶颈** | 节点 > 5000 时渲染卡顿 | 自动启用虚拟化（只渲染视口内节点），提示用户使用过滤缩小范围 |
| **超大项目** | node_modules 未正确排除导致扫描爆炸 | 默认排除模式清单（node_modules / .git / dist / build），固定深度上限 12 层 |
| **依赖解析失败** | 非标准 import 语法（动态 import、路径 alias） | 跳过无法解析的 import，在日志中标注 `[unresolved]`，不阻塞图形渲染 |
| **浏览器不支援** | 旧版浏览器不支援 ES Module 或 D3.js v7 | 显示浏览器兼容性提示，要求使用 Chrome/Firefox/Edge 最新版 |
| **无项目开启** | 用户首次载入页面 | 显示空状态引导画面：图示 + 开启项目按钮 + 简短说明 |
| **LLM API 不可用** | 异步摘要请求失败（无 Key/网络中断） | 静默降级到 F13 硬编码映射，tooltip 使用规则描述而非 LLM 描述 |
| **语义映射 key 缺失** | F13 硬编码映射表中无对应文件的规则 | fallback 到原角色名称（component/service/type），加上 `?` 前缀提示用户可贡献描述 |
| **双视图切换效能** | 过滤基建文件后力导向图需重新布局 | 只隐藏/显示节点，不重启 simulation（re-heat alpha 0.05 微调） |

## 8. 风险与合规声明 (Risk & Compliance)

### 资料隐私边界

- **所有资料本地处理**：Vibe Guarding 不将任何项目文件内容上传至外部服务器。文件路径与内容仅用于本地分析，WebSocket 通讯局限于 localhost
- **无远端通讯**：除载入 D3.js CDN（`d3.v7.min.js`）外，无任何外部网络请求。CDN 载入仅在首次开启页面时发生一次
- **可选断网模式**：D3.js 可切换为本地捆绑载入，实现完全离线执行

### 开源与知识产权

- Vibe Guarding 本身采用 MIT 授权
- 用户在 Vibe Guarding 中载入的 PRD 与项目代码，其知识产权归属于用户自身

### 算法透明度

- 角色辨识与依赖分析均为确定性规则引擎，无不透明决策路径
- 所有推断规则均定义于 `project-analyzer.js` 中，用户可审阅与修改
- 无算法备案要求（不涉及个人资讯处理或自动化决策）
